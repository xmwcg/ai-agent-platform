# 方案①：Serverless（EdgeOne Pages + 云函数 + 国内云 MongoDB）—— 低成本起步

> 适用前提（用户已确认）：**公司主体 + 已有公司微信支付商户号 + 目标市场国内为主 + 要求稳定/专业/闭环**。
> 本方案成本最低（≈¥0–30/月），但**对现有后端有适配改造成本**，需权衡"省钱"与"改造成本/运维复杂度"。
> 闭环定义：**开发 → 部署 → 上线 → 运营 → 免费引流获客 → 付费 → 再获客 → 再付费**。

---

## 0. 本方案一句话结论

前端丢到 **EdgeOne Pages**（国内访问稳、免费额度大），后端拆成**云函数**（腾讯云 SCF / 云开发），数据库用**国内云 MongoDB**（腾讯云/阿里云，规避 Atlas 跨境延迟与备案问题），Redis 用**项目自带的内存降级**（零成本）或 Upstash。可把月现金压到 ≈¥0–30。**代价**：现有后端（MCP stdio 起进程、媒体异步轮询、长连接）需做 Serverless 适配改造，改造工作量与稳定性风险高于方案②。

> ⚠️ 若你追求"最稳、最省事、直接真收款"，**优先看 `PLAN-B-Lightweight.md`（方案②）**。本方案适合"愿意用改造换长期更低月费、且流量有弹性"的场景。

## 3. 必须做的 Serverless 适配改造（★ 本方案核心风险，务必评估）

现有后端（`server/`）是**常驻 Express 进程**，含以下 Serverless 非友好点，需改造：

### 3.1 MCP stdio（本地 spawn 进程）
- **问题**：`/api/mcp` 通过 `child_process` spawn 本地 stdio 服务器，Serverless 函数**无持久进程、无本地文件系统写权限**，无法直接起。
- **改造方案**（任选）：
  - A. MCP 改走 **SSE/HTTP 远程服务器**（不改 spawn，连外部 MCP 服务）；
  - B. 把 MCP 进程托管到**常驻容器/独立服务**（轻量服务器或云容器），函数仅做转发；
  - C. 上线初期**暂不支持 stdio MCP**，仅留 SSE 类型（影响：MCP 插件管理能力受限）。
- **工作量**：中（需改 `services/mcp.service.ts` 连接逻辑 + 配置）。

### 3.2 媒体异步生成（文生视频/图生视频轮询）
- **问题**：`/api/tools/media` 返回 `taskId` 后前端轮询，生成是**长任务**（数秒~分钟），超出函数超时（SCF 默认 3–15s/可调至 60–900s，但按量计费且不稳定）。
- **改造方案**：
  - 长任务拆为**事件驱动**：函数只"提交任务"→ 消息队列/云函数异步 → 结果写库 → 前端轮询库状态。
  - 或把媒体生成放到**独立常驻服务**（同 3.1 的 B 方案），函数仅代理。
- **工作量**：中。

### 3.3 长连接 / WebSocket / 流式对话
- **问题**：AI 流式对话（`/api/ai` sendMessage）在云函数上可用**流式响应**（SCF 支持 SSE 流式），但需确认运行时支持；Redis 会话持久化（内存降级后）OK。
- **改造方案**：对话走 SSE 流式返回；会话状态存 MongoDB（已支持）。
- **工作量**：低–中。

### 3.4 文件上传 / 本地存储
- **问题**：`UPLOAD_DIR=./uploads` 是本地磁盘，函数无持久本地存储。
- **改造方案**：上传改到**对象存储（COS/OSS）** 或直接走 MongoDB GridFS。需改 `services/file*` 与上传中间件。
- **工作量**：低–中。

### 3.5 改造工作量汇总
| 改造项 | 工作量 | 是否阻塞真收款 |
|--------|--------|----------------|
| MCP stdio→SSE/外部 | 中（2–4 天） | 否（可暂屏蔽） |
| 媒体异步→事件驱动 | 中（2–3 天） | 否（可暂用 Mock） |
| 流式对话适配 | 低–中（1–2 天） | 否 |
| 上传→对象存储 | 低–中（1–2 天） | 否 |
| **合计** | **约 1–2 周** | 支付/订阅本身可先行（与方案②同抽象层） |

> **关键结论**：支付/订阅/套餐/配额这些"商业化闭环"**复用同一套代码**（`services/payment.service.ts`、`/api/billing`），Serverless 化**不影响真收款能力**；改造主要针对 MCP/媒体/上传等增强功能的运行形态。可先上线"对话+RAG+知识库+客服+订阅付费"核心闭环，再逐步补 MCP/媒体。

## 4. 门槛打通清单（支付 / 备案 / 质量 / 安全）

> 与方案②本质相同（同一支付抽象层、同一备案要求、同一安全模型），仅运行形态不同。细化如下。

### 4.1 支付（微信商户号联调）
- 复用 `services/payment.service.ts` 的 `WechatGateway`（AES-256-GCM 解密 + RSA 验签）。
- `WECHAT_NOTIFY_URL` 必须公网 HTTPS → 指向云函数 API 网关地址（如 `https://api.你的域名/api/billing/webhook/wechat`）。
- `DEFAULT_PAY_PROVIDER=wechat`、`ENABLE_MOCK_MODE=false`。
- **验收（质量门 G3）**：小额真实支付 → 回调验签 → 会员激活，与方案②一致。

### 4.2 审批 / 备案
- **ICP 备案**：EdgeOne Pages / 云函数绑定**国内域名**需备案（腾讯云免费代办，1–2 周）。
- **经营性 ICP / EDI**：同方案② 4.2，早期可后置，视规模补办。
- 注意：若用 **Vercel**（海外）可跳过备案但**国内访问不稳**，与"国内为主"冲突 → **前端务必用 EdgeOne Pages 而非 Vercel**。

### 4.3 质量（测试 / CI）
- 同方案② 4.3：`npm run test` 全绿、`lint` 0 error、`build` 成功。
- 额外：函数打包后做**本地模拟调用测试**（SCF 本地调试 / 云开发本地），验证无状态、冷启、超时边界。

### 4.4 安全
- 同方案② 4.4：HTTPS（EdgeOne 默认）、强 `JWT_SECRET`（弱值启动拦截）、密钥不泄露、`apiKey` 加密、限流。
- 函数环境**不落密钥到磁盘**，用环境变量 / 云密钥管理（如腾讯云 Secret Manager）。

## 10. 风险与回退
- **函数超时/冷启**：对话流式、媒体异步需事件化（第 3 章）；设成本预算告警。
- **MCP/媒体受限**：上线初期可暂屏蔽 stdio MCP、媒体走 Mock，核心闭环不受影响。
- **回退到方案②**：若 Serverless 改造成本/稳定性不及预期，**代码无需重构**，直接 `docker-compose up -d` 迁到轻量服务器（同一份代码两套部署），这是本平台"可移植"的最大优势。
