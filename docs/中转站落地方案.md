# aibak.site 中转站（大模型聚合网关）落地方案

> 目标：在 **aibak.site 平台项目**下搭建大模型中转站（Relay），统一进货各家大模型、按 License 额度分销给金网通（及未来其他产品），赚取 token 差价并沉淀平台级能力。
> 本方案为**开发/部署计划文档**，代码落地在 aibak.site 项目（仓库 `aibak.site/ai-agent-platform`），与 `enterprise-lan-management`（金网通客户端）解耦。

---

## 1. 定位与边界（白话）

- **aibak.site** = 公司平台门面（公网可访问）。
- **中转站** = aibak.site 下的「供货仓」：统一接好各家大模型，按量出货、按 License 记账收钱。
- **金网通（enterprise-lan-management）** = 派到客户内网的「上门安装师傅」：要 AI 就打电话回总部仓库（填中转站 BaseURL + LicenseId）。
- **绝不**把上游密钥塞进客户机器；中转站单独部署、独立扩容、独立对接支付与备案。

## 2. 架构

## 3. 技术选型

| 选项 | 方案 | 适合度 | 说明 |
|----|----|----|----|
| **A（推荐）** | **New-API 自建** | ★★★★★ | 开源、Docker 一键起、原生 OpenAI 兼容、支持多密钥/渠道轮询/额度/日志；数据自主、可二开计费 |
| B | 聚合平台（302.AI / 硅基流动） | ★★★☆☆ | 省运维，但计费与额度受其限制，需二次封装做 License 绑定 |
| C | 自研轻量网关 | ★★☆☆☆ | 完全可控但开发量大，建议二期再考虑 |

**结论：MVP 用 New-API 自建**，后续如需深度定制计费/订阅再叠加自研层。

## 4. 部署（MVP：New-API on aibak.site 服务器）

### 4.1 前置
- 一台具备公网 IP 的服务器（已有），域名 `relay.aibak.site`（或 `ai.aibak.site`）。
- 安装 Docker + Docker Compose；准备 HTTPS（反向代理 Nginx/Caddy + 证书）。
- 准备各上游 API Key（硅基流动、302.AI、官方直连等）。

### 4.2 docker-compose.yml（示例）
```yaml
version: '3'
services:
  newapi:
    image: calciumion/new-api:latest
    container_name: aibak-relay
    restart: always
    ports:
      - "127.0.0.1:3000:3000"
    volumes:
      - ./data:/data
    environment:
      - TZ=Asia/Shanghai
      - SQL_DSN=/data/newapi.db   # 或接 PostgreSQL
      - SESSION_SECRET=<随机串>
  caddy:
    image: caddy:latest
    restart: always
    ports:
      - "80:80"
      - "443:443"
    volumes:
      - ./Caddyfile:/etc/caddy/Caddyfile
      - caddy_data:/data/caddy
volumes:
  caddy_data:
```
> Caddyfile：`relay.aibak.site { reverse_proxy newapi:3000 }`（自动签发 HTTPS）。

### 4.3 初始化
1. 起服务后访问 `https://relay.aibak.site`，用初始管理员账号登录，改密。
2. 「渠道管理」逐个添加上游（硅基流动 / 302.AI / 官方），设轮询/权重。
3. 「令牌管理」为金网通生成**平台级令牌**；后续按客户 License 维度再细分额度。
4. 验证：`curl https://relay.aibak.site/v1/models -H "Authorization: Bearer <token>"` 返回模型列表。

## 5. 与金网通对接（客户端零改）

金网通 M4 已预留 `我的中转站·智控版` 厂商条目（`lib-llm.ps1`）。对接只需：
1. 在金网通控制台 **⑥** 选「我的中转站·智控版」；
2. `BaseURL` 填 `https://relay.aibak.site/v1`；
3. 鉴权方式二选一（见 §6）；
4. 「获取模型列表」「测试连接」即通。

金网通 `console.html` 的 `relay` 条目 `baseUrl` 当前为占位 `https://relay.your-platform.example/v1`，上线时改为真实域名即可（或直接由平台下发配置）。

## 6. License 额度绑定（计费层，二期核心）

中转站本身做转发，额度/订阅由平台侧绑定：

| 层级 | 做法 | 说明 |
|----|----|----|
| 子密钥 | 每个客户 License 生成独立 token | 被盗仅影响单客户 |
| 额度 | 中转站/平台记录 token 用量，按 License 扣减 | 支持包月/包量 |
| 软控 | 金网通 `relay` 特性门禁（仅 Smart 版开启） | 无 `relay` 特性则客户端隐藏中转条目 |
| 对账 | 中转站用量日志 → 平台账单 | 资金/备案隔离 |

> 可在 New-API 的「令牌额度」之上，再写一层「License → 令牌」映射服务（放在 aibak.site 平台后端），实现真正的按客户订阅计费。

## 7. 里程碑

| 阶段 | 内容 | 产出 |
|----|----|----|
| M1 | New-API 部署 + HTTPS + 上游接入 | 可用的 OpenAI 兼容 BaseURL |
| M2 | 金网通对接验证（⑥ 填中转站跑通 ⑦⑧） | 端到端调通 |
| M3 | License → 令牌映射 + 额度扣减 | 可按客户计费 |
| M4 | 订阅/包量套餐 + 对账后台 | 商业化收银 |

## 8. 安全与合规

- 上游密钥**只存平台侧**；客户机器仅持 LicenseId/子密钥。
- 中转站独立部署、独立备份、独立审计。
- 公网接口强制 HTTPS + 速率限制 + 异常告警。
- 涉及资金/用量，保留对账日志以满足备案与争议处理。

## 9. 后续可扩展

- 多产品复用：aibak.site 其它业务直接接同一中转站。
- 模型路由：按成本/质量自动选上游（New-API 渠道优先级）。
- 私有化：大客户可部署专属中转节点。
