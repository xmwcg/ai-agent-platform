# 小程序引流方案 · 复用现有 billing，零重复造轮子

> 目标：用微信小程序作为「视频号/抖音/小红书」引流的落地承接，把公域流量导入你的付费体系。
> 核心原则：**下单与支付全部复用现有 `/api/billing/*`（Web 已打通微信支付）**，小程序只做「展示 + 跳转」，不写第二套支付。

---

## 一、为什么小程序不下单，只跳转 H5？

你的 `ai-agent-platform` 已经有一套生产级 Web 会员/支付链路：
- `POST /api/billing/orders`（订阅）、`/credits-packages/order`（积分）、`/private-license/order`（企业版）
- 微信支付 v3 Native 回调 `webhook.routes.ts`（已验证）
- 前端 `PricingPage` 已打通扫码支付

**若小程序再写一套微信支付**：
1. 要再申请微信支付「小程序商户号 / 公众号商户号」绑定，重复配置；
2. 回调、对账、积分账本全部双轨，与你现有 `credit-ledger` 冲突（正是之前废弃「独立商城」的教训）。

**正确做法**：小程序作为「引流前端」，购买行为用 `web-view` 打开你的 H5 会员页（`https://aibak.site/pricing`），由现有 Web 链路完成支付。一个商户号、一套账本、零冲突。

---

## 二、小程序定位：引流型（不做重功能）

| 模块 | 内容 | 是否调你后端 |
|------|------|--------------|
| 首页 | 平台价值主张 + 爆款 AI 工具演示（图/视频） | 否 |
| 干货/案例 | 短视频切片、小红书笔记同步展示 | 否 |
| 免费体验 | 跳转 H5 注册页（拿手机号授权 → 你后端建号） | 跳转 H5 |
| 会员/购买 | `web-view` 打开 `/pricing`（订阅+积分+企业版） | 由 H5 完成 |
| 我的 | 跳转 H5 个人中心（订单/额度查询） | 由 H5 完成 |

小程序只保留：**内容展示 + 跳转按钮 + 微信登录（openid 用于绑定你账号）**。

---

## 三、微信登录打通（轻量，可选）

若希望小程序用户与 Web 账号互通，用微信 `wx.login` 拿 code → 调你后端 `POST /auth/wechat/mini-login`（**新增一个极简接口**，仅用 openid 找/建用户并下发 JWT，复用现有 `User` 模型，不碰计费）。

这样用户在小程序点「购买」→ `web-view` 打开 H5 时已带登录态，体验无缝。

> 该接口是「登录绑定」而非「支付」，与现有 billing 完全解耦，安全。

## 四、引流路径（与多平台内容日历联动）

```
视频号/抖音 短视频 ──挂「小程序」链接──> 小程序首页(干货/演示)
                                         │
小红书 笔记 ──置顶评论「搜小程序名」──> 小程序(案例展示)
                                         │
                             点击「立即开通会员」web-view
                                         ▼
                              H5 /pricing(你现有,微信支付)
                                         ▼
                              订阅/积分激活(现有 billing 完成)
```

公众号承接：小程序 `subscribeMessage` 引导关注公众号 → 公众号菜单挂 `/pricing` + 企业微信客服 → 私域复购。

---

## 五、合规红线（微信小程序）

1. 小程序内**不得诱导分享**、不得虚假宣传「AI 能赚钱」类；
2. 虚拟商品（会员/积分）**必须用微信支付**，且必须通过 `web-view` H5 走你已备案商户号（不可小程序内直接卖虚拟币）；
3. AI 生成内容需有「内容安全」过滤（你后端 `/ai/chat` 已有合规层）。

## 六、开发工作量（零资金，单人可交付）

| 项 | 工作量 | 说明 |
|----|--------|------|
| 小程序框架（uniapp 或原生） | 2~3 天 | 首页+干货+我的，纯静态 |

| 审核上线 | 1~3 天 | 微信审核 |

**总计约 1 周可上线首个引流版**，全部复用你现有付费基建，不新增商户、不新增账本。

## 七、是否现在做？

你之前确认「小程序要有方案、对原有项目变现引流有用就做」。结论：**有用，且零冲突**。

建议优先级：**先跑通 Web 会员/企业版（已完成代码改造）→ 再做小程序引流承载**。小程序代码可在你确认后单独建仓库开发，不影响当前 `ai-agent-platform`。
