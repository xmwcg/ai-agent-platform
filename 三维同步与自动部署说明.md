# 三维同步与自动部署操作说明

> 适用项目：ai-agent-platform（AIBAK 前端 + 后端 monorepo）
> 目标：你只推 cnb.cool 一个地方，代码自动镜像到 GitHub、自动部署到服务器上线。

---

## 一、什么是"三维同步"

你的代码在三个地方保持一致：

| 维度 | 是什么 | 作用 |
|------|--------|------|
| 本地电脑 | 你写代码的地方 | 日常开发 |
| CNB（cnb.cool） | `aibak.site/ai-agent-platform` | **唯一入口**：你只推这里 |
| GitHub | `github.com/xmwcg/ai-agent-platform` | 云端备份仓库（由 CNB 自动镜像，无需手动推） |
| 服务器 | `159.75.124.59` 上运行的项目 | 真正给用户访问的地方（已上线） |

---

## 二、真实架构（一条主线 + 一个备用通道）

### 主线：本地 → CNB →（镜像 GitHub + 服务器自动拉取部署）

```
你本地改代码
   ↓ git push origin main  （只推 cnb.cool 一个地方）
CNB 收到 push
   ├─① 自动镜像到 GitHub（.cnb.yml，普通推送，失败才 force-with-lease）
   └─② 服务器自动部署：
         · 主通道：服务器 cron 每分钟轮询 cnb.cool（deploy/cnb-watcher.sh）
         · 备用通道：CNB Webhook 推送触发（webhook-server.py，端口 9000，秒级）
             ↓
          deploy-from-cnb.sh 拉取最新代码 → 构建镜像 → docker compose 上线
             ↓
          健康检查：/api/health 与 /api/sandbox/status 必须返回 200
```

你只管写代码、推一次，`最多 1 分钟内`自动上线（cron 轮询），接了 webhook 后则是`秒级`。

### 关键事实（与旧文档不同）

- **GitHub 是被动镜像**，不是你手动推的，也不是部署入口。
- **真正干部署活的是服务器上的 cron 轮询**（`/etc/cron.d/cnb-watcher`，每分钟一次），不是 webhook。webhook 当前是"已就绪但未在 CNB 后台接通"的备用秒级通道。
- **两条部署路径共用同一把锁和同一个 STATE_FILE**（`/opt/.cnb-deploy.lock` 与 `/opt/.cnb-deploy-sha`），不会重复部署。
- **STATE_FILE 只在"部署成功 + 健康检查通过"后才写**；失败会留 `/opt/.cnb-deploy-failed` 标记，下一分钟自动重试，不会"假成功"卡死。

## 三、部署相关文件（均已纳管进仓库）

| 文件 | 作用 |
|------|------|
| `.cnb.yml` | CNB 流水线：镜像 GitHub（受控强推） |
| `deploy/cnb-watcher.sh` | 服务器 cron 主通道：轮询 + 部署 + 健康检查 + 失败重试 |
| `deploy-from-cnb.sh` | 实际部署脚本：拉代码、构建、上线（被 watcher 和 webhook 共用） |
| `webhook-server.py` | 服务器 Webhook 接收端（端口 9000，HMAC 验签），接通后为秒级部署 |
| `scripts/auto-deploy.cjs` | 本地防抖推送辅助（可选） |

## 四、怎么验证"四端已同步"

在本地跑：

```bash
cd /workspace
echo "LOCAL : $(git rev-parse HEAD)"
echo "CNB   : $(git ls-remote https://cnb.cool/aibak.site/ai-agent-platform.git HEAD | awk '{print $1}')"
echo "GHUB  : $(git ls-remote https://github.com/xmwcg/ai-agent-platform.git HEAD | awk '{print $1}')"
echo "SERVER: $(ssh root@159.75.124.59 'cat /opt/.cnb-deploy-sha')"
```

四个值应一致，且 `https://aibak.site/api/sandbox/status` 返回 200。

## 五、密钥与权限

- **CNB 镜像 GitHub** 用 `.cnb.yml` 里的 `GH_TOKEN`（CNB 仓库变量，不进代码）。
- **服务器部署密钥**：`deploy-from-cnb.sh` 通过环境变量注入（**不再硬编码**在脚本里，避免泄露到公开镜像）。
- 服务器敏感配置（数据库密码、支付密钥、大模型 Key 等）放在服务器 `server/.env`，**不要**提交到仓库。
- 本机用于登录服务器的 SSH 公钥已加入服务器 `/root/.ssh/authorized_keys`（备注 `ai-agent-deploy-bot`），如不再需要可删除该行。

## 七、如何接通"秒级部署"（CNB Webhook，可选）

当前部署靠 cron 每分钟轮询（最多慢 ~1 分钟）。若想 push 即秒级上线，在 **CNB 仓库后台** 配置 Webhook：

1. 打开 cnb.cool → `aibak.site/ai-agent-platform` → **设置 → Webhooks → 新建**。
2. 填写：
   - **URL**：`http://159.75.124.59:9000/deploy`
   - **密钥（Secret）**：与服务器 `webhook-server.py` 中 `SECRET` 环境变量一致（部署密钥，不要公开）。
   - **触发事件**：勾选 `Push`（推送）。
3. 保存后，随便推一次代码测试：服务器 `tail -f /var/log/cnb-webhook.log` 应出现新 POST，部署秒级完成。

> 注意：Webhook 走服务器 9000 端口入站，需确保服务器安全组/防火墙放行该端口（当前 cron 通道不依赖入站，故一直可用）。

## 八、常见问题

**Q：我推了代码，但网站没更新？**
A：先等最多 1 分钟（cron 轮询）。然后上服务器看：
```bash
ssh root@159.75.124.59
tail -f /var/log/cnb-watcher.log      # 主通道日志
cat /opt/.cnb-deploy-sha              # 已部署的 SHA
ls -la /opt/.cnb-deploy-failed        # 若有此文件说明上次部署失败，会自动重试
```

**Q：GitHub 没同步？**
A：看 CNB 流水线 `.cnb.yml` 的 push 阶段是否报错（通常是 `GH_TOKEN` 失效）。GitHub 是被动镜像，不要手动推。

**Q：CI 出现红色 ❌ 要紧吗？**
A：不阻塞上线，但说明代码有测试/类型问题，建议尽快修。

**Q：哪些东西不能进仓库？**
A：密码、支付密钥、API Key、服务器 IP 白名单等留在服务器 `server/.env`；代码和配置可以进。

## 九、手动触发部署（应急）

若需不推代码就重部署：

```bash
ssh root@159.75.124.59
sudo systemctl restart cnb-webhook        # 若走 webhook 通道
# 或直接删除 STATE_FILE 让 cron 下一分钟重新部署：
rm -f /opt/.cnb-deploy-sha
```
