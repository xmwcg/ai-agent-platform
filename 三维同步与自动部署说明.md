# 四端同步与自动部署说明（兼容入口）

当前生产发布的唯一有效方案已经统一到：

- 本地只推 CNB `main`。
- CNB 流水线先执行后端测试、类型检查和前端构建。
- 门禁通过后，安全镜像 GitHub，并晋级 `deploy/production`。
- 生产服务器的 systemd timer 只轮询 `deploy/production`，部署成功且公网验收通过后才记录版本。

完整安装、回滚和验收说明见：[`docs/CNB-FOUR-END-SYNC.md`](docs/CNB-FOUR-END-SYNC.md)。

## 兼容文件

历史上的 `deploy-from-cnb.sh` 和 `webhook-server.py` 仍保留为可选兼容入口，但它们**不能直接部署 `main`**，只会唤醒安全 watcher。生产主通道仍是服务器主动拉取，不要求开放入站端口。

## 秒级部署加速（Webhook，可选）

主通道是服务器 systemd timer 每分钟拉取 `deploy/production`。CNB 流水线在晋级 `deploy/production` 后会额外发一条 POST 到服务器的 `webhook-server.py`（端口 9000），让服务器**立即**检查新版本，实现 push 后秒级上线。

- 该步骤在 `.cnb.yml` 的 `notify-webhook` 阶段实现，**不是**在 CNB 仓库设置页配置。
- `notify-webhook` 是"最佳努力"非阻塞的：即使 POST 失败，systemd timer 也会在 1 分钟内兜底。
- 服务器日志查看方式：
  ```bash
  ssh root@159.75.124.59
  tail -f /var/log/cnb-webhook.log
  tail -f /var/log/cnb-watcher.log
  ```

## 禁止事项

- 不要直接在 GitHub 或生产服务器修改正式代码。
- 不要恢复 GitHub → CNB 的定时反向同步。
- 不要让 webhook 或 cron 直接部署 CNB `main`。
- 不要使用无条件 `git push --force`。
